Requisitos previos

🛠️ Preparar el entorno

  1. Asegurarse de tener el virtualenv activo: source .venv/bin/activate
  2. Verificar que PostgreSQL esté corriendo (local o Neon)
  3. Correr migraciones: python manage.py migrate
  4. Iniciar el servidor de desarrollo: python manage.py runserver
  5. Abrir el navegador en http://localhost:8000
  6. Verificar que la consola muestra emails (configurado con EMAIL_BACKEND = console en local)
💡 En desarrollo local, los emails se imprimen en la terminal donde corre runserver. No se envían realmente.

🔐 Registro de usuario

Registro exitoso (happy path)

Precondiciones: No estar logueado. Servidor corriendo.

  1. Navegar a http://localhost:8000/register/
  2. Verificar que se muestra el formulario con campos: Nombre, Correo, Contraseña, Confirmar contraseña
  3. Completar: Nombre = "Test User", Email = "test@correo.com", Contraseña = "segura123", Confirmar = "segura123"
  4. Click en "Crear cuenta"
  5. Esperado: Redirige a pantalla de verificación pendiente con mensaje "Revisá tu correo"
  6. Verificar en la terminal que se imprimió el email de verificación con un enlace

Registro con email duplicado

Precondiciones: Ya existe un usuario con "test@correo.com" (creado en el test anterior).

  1. Cerrar sesión si estás logueado (o usar otra ventana de incógnito)
  2. Navegar a http://localhost:8000/register/
  3. Completar con el mismo email: "test@correo.com"
  4. Click en "Crear cuenta"
  5. Esperado: Muestra error genérico "No se pudo crear la cuenta" — NO revela que el email ya existe

Registro con contraseña inválida

Precondiciones: No estar logueado.

  1. Navegar a http://localhost:8000/register/
  2. Completar con contraseña "abc" (muy corta, sin número)
  3. Click en "Crear cuenta"
  4. Esperado: Muestra errores de validación (mínimo 8 caracteres, debe tener al menos una letra y un número)
  5. Probar con contraseña "12345678" (solo números)
  6. Esperado: Error de que necesita al menos una letra
  7. Probar con contraseñas que no coinciden
  8. Esperado: Error "Las contraseñas no coinciden"

Registro — usuario logueado es redirigido

Precondiciones: Estar logueado con un usuario verificado.

  1. Iniciar sesión con un usuario verificado
  2. Navegar manualmente a http://localhost:8000/register/
  3. Esperado: Redirige automáticamente a /events/ (no muestra el form de registro)

📧 Verificación de email

Verificación exitosa

Precondiciones: Registrarse con un email nuevo. Tener el enlace de verificación de la terminal.

  1. Registrar un usuario nuevo (email: "verify@test.com")
  2. Copiar el enlace de verificación que aparece en la terminal
  3. Pegar el enlace en el navegador
  4. Esperado: Muestra mensaje "¡Email verificado! Bienvenido a Hatsik." y redirige
  5. Verificar que ahora podés acceder a /events/ sin ser bloqueado

Token ya utilizado

Precondiciones: Haber usado el enlace de verificación del test anterior.

  1. Pegar el mismo enlace de verificación otra vez en el navegador
  2. Esperado: Mensaje "Este enlace ya fue utilizado. Iniciá sesión."

Token inválido

  1. Navegar a http://localhost:8000/verify-email/token-falso-12345/
  2. Esperado: Mensaje "Enlace de verificación inválido." — redirige a login

Reenviar email de verificación

Precondiciones: Estar logueado con un usuario NO verificado.

  1. Registrar un usuario nuevo (no verificar)
  2. En la pantalla de verificación pendiente, click en "Reenviar email de verificación"
  3. Esperado: Mensaje "¡Email de verificación enviado!" y nuevo email en la terminal
  4. Intentar click de nuevo inmediatamente
  5. Esperado: Mensaje "Podés reenviar en 60 segundos" (rate limit)
  6. Esperar 60 segundos y volver a intentar
  7. Esperado: Funciona de nuevo

🔑 Inicio de sesión

Login con usuario verificado

Precondiciones: Tener un usuario verificado creado. No estar logueado.

  1. Navegar a http://localhost:8000/login/
  2. Verificar que se muestra el formulario con campos: Correo electrónico, Contraseña
  3. Ingresar credenciales válidas del usuario verificado
  4. Click en "Iniciar sesión"
  5. Esperado: Redirige a /events/

Login con usuario NO verificado

Precondiciones: Tener un usuario registrado pero sin verificar.

  1. Navegar a http://localhost:8000/login/
  2. Ingresar credenciales del usuario no verificado
  3. Click en "Iniciar sesión"
  4. Esperado: Redirige a /verify-email/ (pantalla de verificación pendiente)
  5. Verificar que NO puede acceder a /events/ — el middleware lo bloquea

Login con credenciales inválidas

  1. Navegar a http://localhost:8000/login/
  2. Ingresar email correcto pero contraseña incorrecta
  3. Click en "Iniciar sesión"
  4. Esperado: Error genérico "Correo o contraseña incorrectos" — NO indica cuál campo es incorrecto
  5. Ingresar un email que no existe en el sistema
  6. Esperado: Mismo error genérico (no revela si el email existe)

Parámetro ?next= preservado

  1. Sin estar logueado, navegar a http://localhost:8000/login/?next=/events/123/
  2. Ingresar credenciales válidas de un usuario verificado
  3. Esperado: Después del login redirige a /events/123/ (no a /events/)

🚪 Cierre de sesión

Logout exitoso

Precondiciones: Estar logueado.

  1. Desde cualquier pantalla autenticada, realizar logout (botón o POST a /logout/)
  2. Esperado: Redirige a /login/
  3. Intentar navegar a /events/
  4. Esperado: Redirige a login (sesión destruida)

Logout requiere POST

  1. Estar logueado
  2. Navegar con GET a http://localhost:8000/logout/
  3. Esperado: Error 405 (Method Not Allowed) — no se puede hacer logout con GET

🔄 Recuperar contraseña

Solicitar reset — email existente

Precondiciones: Tener un usuario registrado. No necesita estar logueado.

  1. Navegar a http://localhost:8000/password-reset/
  2. Ingresar el email del usuario existente
  3. Click en "Enviar enlace"
  4. Esperado: Mensaje genérico "Si este correo existe en nuestro sistema, enviamos un enlace..."
  5. Verificar en la terminal que se imprimió el email con enlace de reset

Solicitar reset — email que NO existe

  1. Navegar a http://localhost:8000/password-reset/
  2. Ingresar un email que no existe: "noexiste@fake.com"
  3. Click en "Enviar enlace"
  4. Esperado: MISMO mensaje genérico (no revela que el email no existe)
  5. Verificar en la terminal que NO se imprimió ningún email

Restablecer contraseña con token válido

Precondiciones: Haber solicitado un reset y tener el enlace de la terminal.

  1. Copiar el enlace de reset de la terminal
  2. Pegarlo en el navegador
  3. Esperado: Formulario para nueva contraseña
  4. Ingresar nueva contraseña: "nuevapass1" en ambos campos
  5. Click en "Restablecer contraseña"
  6. Esperado: Mensaje de éxito y redirige a login
  7. Iniciar sesión con la nueva contraseña
  8. Esperado: Login exitoso

Rate limit en solicitud de reset

  1. Solicitar reset para un email
  2. Inmediatamente solicitar de nuevo para el mismo email
  3. Esperado: Muestra el mismo mensaje genérico (pero NO envía segundo email — verificar en terminal)

🛡️ Protección de rutas

Usuario anónimo bloqueado

  1. Sin estar logueado, navegar a http://localhost:8000/events/
  2. Esperado: Redirige a /login/?next=/events/
  3. Probar con cualquier otra ruta protegida
  4. Esperado: Siempre redirige a login

Usuario no verificado contenido

Precondiciones: Logueado con usuario no verificado.

  1. Iniciar sesión con un usuario que NO verificó su email
  2. Intentar navegar a /events/
  3. Esperado: Redirige a /verify-email/
  4. Intentar navegar a cualquier otra ruta protegida
  5. Esperado: Siempre redirige a verificación — NO puede hacer nada más
  6. Verificar que SÍ puede acceder a: /verify-email/, /logout/, /password-reset/

Usuario verificado pasa libremente

Precondiciones: Logueado con usuario verificado.

  1. Iniciar sesión con un usuario verificado
  2. Navegar a cualquier ruta (/events/, etc.)
  3. Esperado: Accede sin problemas (puede dar 404 si la vista no existe aún, pero NO redirige a verify-email)

⚠️ Casos límite

Token de verificación expirado

Para probar este caso, necesitás modificar temporalmente el TTL del token en views.py cambiando VERIFICATION_TOKEN_TTL = timedelta(hours=24) a timedelta(seconds=5), registrar un usuario, esperar 5 segundos, y usar el enlace.
  1. Modificar el TTL temporalmente (ver callout arriba)
  2. Registrar un usuario nuevo
  3. Esperar a que expire el token
  4. Hacer clic en el enlace de verificación
  5. Esperado: Mensaje "Este enlace de verificación expiró. Solicitá uno nuevo."
  6. Revertir el cambio de TTL

Email normalizado (mayúsculas/minúsculas)

  1. Registrar con email "TEST@Example.COM"
  2. Esperado: Se guarda como "test@example.com"
  3. Intentar login con "TEST@Example.COM"
  4. Esperado: Funciona (se normaliza antes de buscar)

Múltiples pestañas / sesión concurrente

  1. Abrir dos pestañas: una en /login/ y otra en /register/
  2. Registrar en una pestaña
  3. En la otra pestaña intentar ir a /events/
  4. Esperado: Redirige a /verify-email/ (la sesión se creó en el registro)

CSRF protection

  1. Abrir DevTools → Network
  2. Enviar un form de login/registro
  3. Verificar que el request incluye el token csrfmiddlewaretoken
  4. Intentar hacer un POST desde curl sin el token CSRF:
  5. curl -X POST http://localhost:8000/login/ -d "email=x&password=y"
  6. Esperado: Error 403 Forbidden (CSRF verification failed)

✅ Checklist final — Done Criteria del ROADMAP

Estos son los criterios de aceptación definidos en docs/ROADMAP.md para la Fase 2.

  • El usuario puede registrarse, recibir email de verificación, hacer clic en el enlace y acceder a la app.
  • El usuario no verificado está bloqueado de todas las funcionalidades y ve la pantalla de Verificación Pendiente.
  • Reenviar verificación respeta rate limit de 60 segundos.
  • El usuario puede iniciar sesión con cuenta verificada.
  • Login de cuenta no verificada redirige a pantalla de verificación.
  • Credenciales inválidas muestran error genérico (no revela existencia del email).
  • Recuperación de contraseña envía email y permite restablecer.
  • Tokens expirados muestran error apropiado.
  • Las 4 pantallas Stitch (Registro, Verificación, Login, Recuperar Contraseña) están implementadas con los tokens del design system.
